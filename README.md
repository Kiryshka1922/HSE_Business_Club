# HSE Business Club — мини-приложение для знакомств и мероприятий

Проект представляет собой веб-приложение для студентов бизнес-клуба ВШЭ, состоящее из двух основных разделов:
- **Dating** — карточки людей с возможностью лайка/дизлайка
- **Schedule** — расписание мероприятий

Приложение построено по современному стеклу с использованием **Feature‑Sliced Design (FSD)** на фронтенде, **NestJS** на бэкенде и **MongoDB** в качестве базы данных. Вся инфраструктура контейнеризована с помощью Docker Compose.

---

## 🛠 Технологический стек

### Frontend
- **React 19** с TypeScript
- **React Router v7** – маршрутизация с SSR-поддержкой (рендеринг на стороне сервера)
- **Zustand** – управление состоянием
- **Vite** – сборка и dev‑сервер
- **CSS‑modules** с методологией **BEM** (стилизация)
- **Stylelint** – линтинг CSS с правилами порядка свойств и BEM‑паттернами
- **Zod** – валидация схем (генерация из OpenAPI)

### Backend
- **NestJS** – фреймворк для построения масштабируемых серверных приложений
- **MongoDB** + **Mongoose** – хранение данных
- **Swagger (OpenAPI 3)** – автоматическая генерация документации API
- **Class‑validator / class‑transformer** – валидация входящих DTO
- **Docker** – контейнеризация

### Инфраструктура
- **Docker Compose** – оркестрация сервисов (frontend, backend, MongoDB)
- **PNPM** – менеджер пакетов
- **Git** – контроль версий

## 🏗 Архитектура

### Frontend – Feature‑Sliced Design (FSD)
Структура фронтенда следует методологии FSD, что обеспечивает чёткое разделение ответственности и масштабируемость:

```
src/
├── app/                    # Инициализация приложения
│   ├── layouts/           # Общие макеты
│   ├── routes/            # Конфигурация маршрутов
│   └── styles/            # Глобальные стили, шрифты, сброс
├── pages/                 # Страницы (фичи)
│   ├── DatingPage/        # Страница знакомств
│   │   ├── lib/           # Логика (стор)
│   │   └── ui/            # Компоненты страницы
│   └── EventsPage/        # Страница мероприятий
└── shared/                # Переиспользуемые модули
    ├── api/               # API‑клиент (axios)
    ├── schemas/           # Zod‑схемы (сгенерированы из OpenAPI)
    └── ui/                # Общие компоненты (GradientBox и др.)
```

### Backend – модульная архитектура NestJS
Бэкенд разделён на доменные модули:

- **DatingModule** – управление данными людей (CRUD)
- **ScheduleModule** – управление мероприятиями

Каждый модуль содержит:
- **Controller** – обработка HTTP‑запросов
- **Service** – бизнес‑логика
- **DTO** – объекты передачи данных с валидацией
- **Schema (Mongoose)** – модель MongoDB

### Стили – BEM + CSS‑modules
Именование классов следует методологии BEM (Block‑Element‑Modifier).  
В проекте настроен **Stylelint** с плагином `stylelint‑selector‑bem‑pattern`, который автоматически проверяет соответствие классов BEM‑нотации.  
Свойства внутри CSS‑файлов упорядочены по группам (позиционирование, отображение, размеры, типографика и т.д.) для единообразия.

### Анимации и визуальные эффекты
В интерфейсе используются:
- Плавные переходы (`transition`) для интерактивных элементов (кнопки, карточки)
- Сложные фоновые элементы с `clip‑path` (ромбы, градиенты)
- Градиентные подложки для создания глубины
- Анимировал path для svg, получилось неплохо

## 🚀 Инструкция по запуску

### Предварительные требования
- **Docker** и **Docker Compose** (для запуска через контейнеры)
- **Node.js 20+** и **PNPM** (если запускаете без Docker)

### 1. Запуск через Docker Compose (рекомендуется)
Бек, база и фронт поднимаются с помощью одной команды:

```bash
cd HSE_Business_club
docker-compose up --build
```

После сборки контейнеров приложение будет доступно:
- **Frontend**: [http://localhost:8080](http://localhost:8080)
- **Backend API**: [http://localhost:3000/api](http://localhost:3000/api)
- **Swagger документация**: [http://localhost:3000/api](http://localhost:3000/api) (интерактивный UI)
- **MongoDB**: `localhost:27017` (база `mock`)

### 2. Локальный запуск (без Docker)
#### Backend
```bash
cd backend
pnpm install
pnpm start:dev
```
Сервер запустится на порту 3000, автоматически заполнит базу моковыми данными и сгенерирует `openapi.yaml`.

#### Frontend
```bash
cd frontend
pnpm install
pnpm run dev
```
Фронт будет доступен на [http://localhost:5173](http://localhost:5173).

### 3. Генерация Zod‑схем из OpenAPI
При изменении API‑контракта бэкенда можно обновить типы фронтенда:

```bash
cd frontend
pnpm openapi-to-zod
```
Скрипт читает `../backend/openapi.yaml` и генерирует файл `src/shared/schemas/schemas.ts` с Zod‑схемами.

## 🔗 API Endpoints

### Dating
- `GET /api/peoples` – список всех людей
- `PATCH /api/peoples/:id` – обновление данных

### Schedule
- `GET /api/events` – список мероприятий
- `PATCH /api/events/:id` – обновление

Детальная документация доступна в Swagger UI после запуска бэкенда.

## 🎨 Особенности реализации

### Рендеринг на стороне сервера (SSR)
На фронте использую **React Router v7** с встроенной поддержкой SSR. Это улучшает SEO и время первой отрисовки.

### Централизованное состояние (Zustand)
Для управления состоянием страниц (например, список людей, лайки) применяю легковесный стейт-менеджер Zustand. Сторы находятся в `pages/*/lib/`.

### Валидация данных
- На бэкенде: DTO с декораторами `class‑validator`
- На фронтенде: Zod‑схемы, сгенерированные из OpenAPI, гарантируют типобезопасность при работе с API.

### Code‑style и линтинг
- **ESLint** (backend) + **Prettier** – единый стиль кода
- **Stylelint** (frontend) – порядок CSS‑свойств и BEM‑валидация
